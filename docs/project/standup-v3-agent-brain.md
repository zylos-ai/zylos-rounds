# 语音日报 v0.3.0 — 给语音 Agent 一个可维护的「大脑」

> 目标读者：Howard。这份是总方案，讲清楚**为什么这么设计**和**边界**，
> 具体实现细节在代码与 CHANGELOG 里。对应 Howard 2026-07-18 的三点期望。

## 1. 背景与目标

语音日报已经能跑通「成员用专属链接和 Luna 语音汇报 → 自动生成结构化日报」。
Howard 希望它从一个「采集器」进化成一个**越用越称手的助手**，提出三件事：

1. 给语音 agent 一份**背景信息容器**，我或 coco 分身能方便地维护。
2. 让 agent 在汇报过程中对 Howard 关心的点**智能追问**——但它怎么知道
   *什么时候问、问什么、问到什么程度*？
3. 把这个应用做成**信息的载体**：原始对话与汇总都存下来备查；agent 在对话中
   判断需要时，能**调接口去取**团队知识 / 记忆 / 过去的对话。

一句话总结这版的设计思想：**agent 的行为由「可编辑的内容」驱动，而不是写死在代码里。**
内容随时可调，每次通话即时生效——这就是「越用越称手」的飞轮。

## 2. 设计总览

把 agent 的「大脑」拆成三种能力 + 一个账本 + 一个维护面：

| 能力 | 机制 | 对应 Howard 的诉求 |
|------|------|--------------------|
| **背景（Push）** | 每次通话把背景容器注入 agent 指令 | ① 背景容器 |
| **追问（Instinct）** | 「追问指引」容器 = 什么时候/问什么/问多深 | ② 智能追问 |
| **记忆（Pull）** | agent 按需调用两个检索工具 | ③ 调接口取信息 |
| **账本（Ledger）** | 原始对话 + 结构化汇总，后台可查 | ③ 存下来备查 |
| **维护（Loop）** | 带 token 的管理接口 + 后台页面 | ①②③ 我/分身维护 |

### 2.1 背景（Push）——总是注入

三个可编辑的背景容器，通话开始时组合进 agent 的 system 指令：

- **团队背景**（全局）：团队在做什么、当前重点。帮 agent 听懂同事在说什么。
- **每个成员的背景**（按人）：这个人的角色、当前项目、需要重点关注的点。
- **追问指引**（全局）：见下。

只有非空的容器才会注入，未配置的安装保持干净。

### 2.2 追问（Instinct）——回答「怎么知道要追问」

这是这版的核心。Howard 的问题「它怎么知道什么情况下要追问、追问什么、追问到
什么程度」——答案就是**「追问指引」这个容器本身**。它不是写死的规则，而是
一段你能随时改的自然语言指引，例如默认种子：

> - 当对方说某件事「差不多做完/基本完成」时，追问一句是否已经验证过、怎么验证的。
> - 当对方提到卡点或风险时，追问一句需要谁配合、卡了多久。
> - 对方已经说得很具体，就不要为了追问而追问。

你跟我或分身说「我关心 X」→ 我们把它写进追问指引 → 下一次通话 agent 就照做。
（旧版写死的「最多追问一句」被这个可编辑指引取代。）

> 为什么追问用 LLM 判断而不是写代码：追问是语言与情境判断，属于「自然」类任务，
> 用模型合适；而检索匹配是确定性任务，用代码（见 2.3）。工具匹配任务的本质。

### 2.3 记忆（Pull）——agent 按需调接口

给 agent 两个 realtime 工具，它自己判断什么时候调用：

- **recall_member_history**：这位同事最近几次的日报（昨天/今天/卡点/待议题）。
  用于跟进——「上次你说要做 X，做完了吗？」让日报有连续性。数据本来就在库里。
- **search_team_knowledge**：对团队知识库做关键词检索（AND 语义 + 标题加权，
  纯代码匹配，不用 LLM）。用于 agent 遇到某个项目/名词需要背景时。

「可以让它取什么」= 成员过去的汇报（记忆）+ 团队知识库（知识）。都由我/分身维护。

### 2.4 账本（Ledger）——存下来备查

- 原始对话（transcript）和结构化汇总本来就落库；这版把**原始对话搬到后台可查**：
  日报页每位成员卡片下有「原始对话」，点开看完整逐字对话。
- 汇总与原文一份都不少，都能翻旧账。

### 2.5 维护（Loop）——我和分身怎么调

大脑内容有两条维护通道：

- **人**：后台新增「背景/追问」页（团队背景 + 追问指引 + 知识库），成员管理页
  每个人加了「背景」编辑入口。
- **我 / coco 分身**：带 **bearer token** 的管理接口（`config.serviceToken`，
  首次启动自动生成），跟 recruit 一样的模式。我和分身能直接用脚本调，不用登录。
  接口清单见 SKILL.md「Maintenance API」。

安全边界：service token 只能改**大脑内容**（背景/追问/知识/成员背景），
**不能**管成员名册（增删成员、看/重置专属链接仍只走管理员登录）。

## 3. 数据模型（migration v3）

- `agent_context(key, value)` — 全局容器（team_background / probing_guidance）
- `members.context` — 每人的背景（可空）
- `knowledge(id, title, content, tags, updated_at)` — 团队知识库

不改任何已有表结构，纯增量迁移。

## 4. 边界与后续

**这版做了**：可维护的背景/追问/知识 + 两个检索工具 + 原文可查 + 带 token 的
维护接口 + 后台 UI。全部有单测（26 个）与本地端到端验证。

**这版没做（可后续）**：
- 知识库现在是关键词检索；条目多了可升级为向量检索。
- 检索工具目前限成员自己的历史 + 团队知识库；未来可接入更多来源（memory、
  过往跨成员对话）——先控制边界，用起来再加。
- 「追问到什么程度」目前靠指引 + 模型判断；如果需要更强的一致性，可以引入
  按点的结构化追问清单。

## 5. 一句话交付

语音日报现在有了一个你和分身随手就能调的「大脑」：想让它更懂团队、更会追问、
记得住事——改内容就行，下一次通话立刻见效。
